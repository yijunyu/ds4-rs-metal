# 在 C 参考实现的主场击败它：用 Rust 在 Apple Silicon 上跑 DeepSeek‑V4‑Flash

*一个 [antirez](https://github.com/antirez) 的 `ds4` 引擎的 Rust + Metal 移植，如何在
Mac Studio 上把 prefill 跑到约 300 tok/s——以及它教会我们的「如何喂饱 GPU」。*

---

## 先致敬

这篇文章首先是一封写给 **Salvatore Sanfilippo（[antirez](https://github.com/antirez)）**
的感谢信。

当他发布 **`ds4`**（"DwarfStar"）时，他做了那件又难又不起眼的事：他用一份小巧、易读、
MIT 许可的 C 引擎，在**单台 Apple Silicon 机器**上把一个前沿级的混合专家（MoE）模型
**DeepSeek‑V4‑Flash** 跑得飞快。没有集群，没有 Python 的层层叠叠，只有 C、Metal，以及
对模型的深刻理解。本文的每一个数字都是**对照他的引擎**测出来的，而我们引擎在 prefill 路径
上跑的大多数 Metal kernel 就是**他的 kernel，原样照搬**，依据他选择的 MIT 许可使用。仓库
里的 provenance 文件精确标注了哪些 shader 是他的。

我们并不是靠在数学上更聪明才胜过 antirez。我们只是在一个特定的点上、靠对**如何喂饱 GPU**
更加讲究，把*时钟*抢回了一截。这就是那一截的故事。

## 机器与模型

- **硬件：** Apple **M1 Ultra**，**128 GB** 统一内存，Mac Studio。
- **模型：** **DeepSeek‑V4‑Flash**，用 iq2 imatrix 量化到 **约 86 GB**。这是个很紧的
  装载——光模型就占了机器内存的三分之二——而这恰恰让调度变得关键。
- **参考：** antirez 的 `ds4`，同机器、同 GGUF、同上下文长度。

DeepSeek‑V4‑Flash 不是玩具。它有多头潜在注意力（MLA）、带数百个专家的稀疏 MoE 前馈、
一条压缩的长程记忆路径，以及一个滑动窗口（SWA‑128）的原始注意力带。prefill——吞进一个
3000‑token 的 prompt——是一长串巨大的矩阵乘和注意力 kernel，深达数千次 GPU 命令缓冲
（command buffer）提交。

## 核心结论

| 阶段（ctx = 3000） | antirez `ds4`（C） | ds4-rs-metal（Rust） | Δ |
|---|---|---|---|
| **Prefill** | ~255 tok/s | **~300–304 tok/s** | **+18–19 %** |
| **Decode** | ~22 tok/s | ~22 tok/s | 持平 |

两条坦诚的前置说明——因为一个不可信的基准测试比没有基准测试更糟：

1. prefill 的数字是**各自在其忠实默认配置下**对比的。我们默认跑模型训练好的
   **压缩长程 + SWA‑128** 路径；antirez 的参考实现在该上下文下跑全原始（all‑raw）。这是
   一个真实的端到端*用户吞吐*差异——是你实际会看到的——但它**不是**「我们算的 FLOP 更少」
   的主张。在完全相同的 all‑raw 模式下，我们与之持平或略微落后。算术还是同一份算术。

2. 本文余下篇幅要讲的是一个**独立且更强**的结果：一个**与基线逐位完全一致**的
   **+13.6 % prefill 提速**（logit 余弦 = 1.0，最大差 = 0.0，在 ctx 600 *和* 3000 都如此）。
   没有牺牲精度，没有改变任何一个 token。这个结果在**任何**配置下都成立，纯粹是 GPU 调度。

## 那个「气泡」

这是我们盯了很久的东西。

当你把 prefill 跑成一串 Metal **命令缓冲**——提交一块工作、等它、再提交下一块——GPU 其实
并不是全程都忙。我们测过：在我们尝试的*每一种*提交策略下，GPU 自身的繁忙时间都
**稳稳地停在 10.46 s**。但 wall‑clock 是约 11.8 s。那约 1.3 s 的差距——大约 **11 %**——
就是 GPU 在命令缓冲*之间*空转，约 21 次、每次约 62 ms，等着 CPU 注意到上一块缓冲已完成
并排上下一块。

我们叫它「气泡」（the bubble）。在大半个开发周期里，我们都以为它是**不可消除的**——一种
每个命令缓冲固有的 flush 延迟。我们试了那些显而易见的办法，全都失败了：

- **用流水线式提交把 CPU 编码重叠起来？** 没用。CPU 不是瓶颈。
- **完全异步、永不阻塞？** 灾难——53 tok/s。在 86 GB 权重被钉住的情况下，让命令缓冲堆积
  把驻留（residency）搅得天翻地覆，GPU 繁忙时间从 10 s *膨胀*到 54 s。
- **打包更大的命令缓冲（更少、更胖的提交）？** 大缓冲的开销吃掉了节省。
- **跨块共享编码器？** −0.9 %。噪声。

这些办法每一个都把气泡当成 *CPU* 问题或*打包*问题。它两者都不是。

## 从 GPU 这一侧喂饱 GPU

最终的解法，是彻底不再去想 CPU。

气泡之所以存在，是因为每个命令缓冲的完成都是在**主机端**被观察到的，然后才排下一块。GPU
干完、举手、等 CPU 把下一页递给它。如果下一页早就摆在 GPU 的桌上、被安排成在当前这块一完成
的瞬间就开始——而 CPU 全程不掺和呢？

这就是 **`MTLEvent`**。我们还是像之前那样把 prefill 切成命令缓冲，但每块缓冲在完成时
**发出（signal）**一个 event，下一块缓冲则**在 GPU 自己的时间线上等待（wait）**这个 event。
下一块被预先排好，在 GPU 上紧随当前块之后。GPU 一块接一块地连续流动；主机不再是节拍器。

结果：

- GPU **繁忙 / 跨度** 从 **88 % 升到 100 %**。
- 气泡从 **1.3 s 坍缩到 1.1 ms**。
- prefill 从 **247 → 280 tok/s**（+13.6 %）——而且 logit **完全一致**。我们用余弦/最大差
  的测试夹具（而不是有噪声的 argmax）验证，cos = 1.0、max‑diff = 0.0，在短上下文和满上下文
  都如此。

有一个细节留下了它自己的伤疤。这个想法的一个早期版本曾被**回退**，因为它在内存压力下崩了
——在一个 77 GB 的后台守护进程下它开始吐零。原因不在 event；而在于那些在飞的命令缓冲各自
持有**全新的、非驻留的临时（scratch）内存**，系统一紧张就缺页成零。解药是一个**有界的、
钉住的 scratch 池**——O(1) 个常驻、复用的缓冲——外加一个**窗口为 1**（在飞的命令缓冲永远
不超过两块）。池钉住、窗口有界之后，这个收益既真实*又*抗压。当初的回退是对的；它只是少了
那个池。

> 我们在这套硬件上反复重学的教训：**空转的 GPU 几乎从来不是 GPU 的问题**，而是喂料的问题。
> 确定性的硅做确定性的工作；如果它空着，是*你*让它空着的。

## 再白捡两个百分点

气泡消失后，prefill 变成了真正的 **GPU 计算受限**——繁忙/跨度 100 %。于是下一个杠杆必须
去掉*工作量*，而不是去掉等待。

MoE 路由器派发的 grid 是按「任一单个专家可能收到的最坏 token 数」来定尺寸的。参考实现很保守
（`ceil(K/4)`）。但这个模型的路由器是无辅助损失的*均衡*型：在一次 3000‑token 的 prefill 里，
观测到的单专家最大 token 数是 187，而 `ceil(K/8)` 已经覆盖 375——**2 倍的安全余量**。把 grid
从 `K/4` 收紧到 `K/8`，去掉了过度派发，prefill 从 **295.6 → 304.3 tok/s**（+3 %），仍然
**逐位一致**。我们就停在这里：`K/16` 会更快但零余量，而为了两个百分点去冒一个静默丢 token
的风险并不值得。

这就是堆叠出来的 **+18–19 %** 核心数字：调度收益，加上更便宜的训练长程路径，加上 grid
收紧。而那个坦诚的、与配置无关、无需任何附注的数字，是 **+13.6 % 逐位一致**的调度结果。

## 为什么用 Rust，以及 kernel 从哪来

引擎是 Rust 写的。MLA + MoE + 注意力的编排、驻留管理、event 流水线调度器、有界 scratch 池
——全是 Rust，没有 GC，对内存*何时*移动没有意外。**Metal kernel** 则是混合的：prefill 路径
上是 antirez 的原样 shader（他的实现体，MIT，在 `PROVENANCE.md` 中标注），外加由
**[tile-rs](https://github.com/yijunyu/tile-rs)**（我们的 Rust kernel 代码生成框架）发出的
kernel——它的 **[Metal 后端](https://github.com/yijunyu/tile-rs-metal)** 把一个形状受检的
tile DSL 下降为 MSL。那一层代码生成已经开源。

## 我们*不*主张什么

- 我们**并不**比 antirez 更懂这个模型。我们是*从*他的引擎里学会这个模型的，他的 C 很出色。
- 我们**没有**算更少的 FLOP。同样的 MLA、同样的 MoE、同样的数学。
- 我们**没有**在 decode 上取胜；那是平手。
- 核心数字里最大的单一贡献者是**调度**，而那一项我们能逐位捍卫。

我们*会*主张的东西很窄、也很真：**在一台 128 GB 的 Mac Studio 上，你可以用一个 Rust 引擎跑
DeepSeek‑V4‑Flash，并拿到明显高于 C 参考实现的 prefill 吞吐，而且最要紧的那部分输出逐位
一致**——因为我们在喂饱 GPU 这件事上更讲究了。
